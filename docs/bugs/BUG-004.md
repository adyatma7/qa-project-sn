# BUG-004: REST API login rejects valid credentials that UI login accepts

**Severity:** High
**Priority:** P1
**Environment:** Chromium via Playwright, ParaBank public demo, 2026-07-2x

## Steps to Reproduce
1. `GET /parabank/services/bank/login/john/demo`
2. In the same session, shortly after: log in via the UI form at
   `/parabank/index.htm` with username `john`, password `demo`

## Expected Result
Both authenticate successfully — same account, same password, same
approximate moment. If one succeeds, the other should too.

## Actual Result
The REST API call returns `400 Bad Request`, body: `Invalid username
and/or password`. The UI login, run immediately after with the identical
credentials, succeeds and reaches Accounts Overview normally.

## Evidence
Reproduced twice, in the same order, close together in time:
1. CI run: API test failed with 400; UI login test passed in the same
   run.
2. Local re-run, specifically to check this: API call failed with the
   identical 400 message; the UI login test, run immediately after in
   the same session, passed cleanly.

Full investigation trail, including the reasoning before this was
confirmed: `docs/bugs/OBSERVATION-003.md`.

## Notes
- **Two authentication code paths, one apparently more strict (or
  broken) than the other.** The REST login service and the UI login form
  most likely don't share the same underlying validation logic — this is
  the kind of inconsistency that matters for anyone building a client
  against ParaBank's documented API, even though the demo UI itself works
  fine.
- **Not the same bug as BUG-001**, and possibly the opposite shape of
  problem: BUG-001 is the UI login *accepting* a wrong password
  (fails open); this is the API *rejecting* a right one (fails closed).
  Worth stating plainly that these are different failure directions, not
  assumed to share a root cause without evidence.
- **Regression test:** `test_valid_login_returns_customer_via_api` keeps
  asserting the correct expected behavior (successful login) — not
  rewritten to expect failure — and is marked `xfail(strict=False)`,
  consistent with how BUG-001/002/003 are handled.
- **Root cause: unconfirmed, no server access.** Could be a bug in the
  REST service's own credential check, a session/state inconsistency
  specific to this heavily-shared account, or something else entirely.
  Stated as unknown rather than guessed.
