# BUG-001: Invalid password still authenticates the user (auth bypass)

**Severity:** Critical
**Priority:** P0
**Environment:** Chromium via Playwright, ParaBank public demo, 2026-07-16

## Steps to Reproduce
1. Go to `https://parabank.parasoft.com/parabank/index.htm`
2. Username: `john`
3. Password: any incorrect value (e.g. `definitely-not-the-real-password`)
4. Click "Log In"

## Expected Result
Login rejected. Page shows: "The username and password could not be
verified." User stays unauthenticated, on the login page.

## Actual Result
User is redirected to `overview.htm` (Accounts Overview) **fully
authenticated** — "Welcome John Smith," real account balance and number,
working "Log Out" link. At the same time, a `.error` banner reads: "An
internal error has occurred and has been logged." — suggesting the
password-verification step itself throws a server-side exception, and
whatever handles that exception fails **open** (lets the user in) instead
of failing **closed** (rejecting the login).

## Evidence
Reproduced twice, back to back, via
`playwright-tests/tests/ui/auth/test_login.py::test_invalid_password_shows_error`.
Same symptom shape both times. First occurrence showed an unrelated
oddity (account display name was garbled — see `OBSERVATION-001.md`,
now superseded by this file); second occurrence showed the correct name
"John Smith," ruling that out as the explanation and confirming this is
the app's real, repeatable behavior — not shared-account interference.

## Notes
- **Regression test already exists** and is marked `xfail(strict=True)`
  rather than deleted or rewritten to match the buggy behavior — it keeps
  asserting what *should* happen. `strict=True` means the day this bug
  gets fixed, the test flips to an unexpected pass and fails the build on
  purpose, which is the signal to remove the marker.
- **Security implication, stated for portfolio purposes** (this is a public
  demo, not a real bank): if this reflects genuine application logic, it's
  an authentication bypass — a known username can log in without the
  correct password whenever this internal error path triggers.
- **Root cause: unverified, stated as a hypothesis only** — no server access
  to confirm. Most likely explanation: an exception during password
  verification is caught by error-handling code that defaults to treating
  the session as valid instead of rejecting it. Not confirmed.
