# BUG-003: Bill Pay does not enforce a sufficient-balance check

**Severity:** High
**Priority:** P1
**Environment:** Chromium via Playwright, ParaBank public demo, 2026-07-19

## Steps to Reproduce
1. Log in as `john` / `demo`
2. Go to `billpay.htm`
3. Fill in valid payee details, amount: `999999999`
4. Click "Send Payment"

## Expected Result
Payment rejected — amount far exceeds any realistic account balance.

## Actual Result
"Bill Payment Complete" is shown. The payment is accepted regardless of
the account's actual balance — same symptom as
[BUG-002](BUG-002.md) on Fund Transfer.

## Evidence
`playwright-tests/tests/ui/bill-pay/test_bill_pay.py::test_bill_pay_amount_exceeds_balance`,
using the same settled-page-then-check assertion pattern already
established after DEC-012. Clean, unambiguous failure — full page
snapshot confirms a genuinely rendered
`<h1 class="title">Bill Payment Complete</h1>`.

## Notes
- **Likely the same root cause as BUG-002, not a coincidence.** Two
  independent money-moving features (Transfer, Bill Pay) both skip
  balance validation — more consistent with a missing validation layer
  shared across ParaBank's payment features than with two unrelated
  bugs. Filed as a separate report because the reproduction steps and
  UI are genuinely different, but the two should be read together.
- **Origin:** this specific test case came from a documented AI-assisted
  brainstorming exercise (`ai-assisted-testing/bill-pay-negative-cases.md`),
  kept specifically because BUG-002 made "does Bill Pay have the same
  gap" the obvious next question — this is the payoff of that reasoning,
  not a lucky guess.
- **Regression test:** `test_bill_pay_amount_exceeds_balance` marked
  `xfail(strict=False)`, consistent with BUG-002's treatment.
- **Not yet cross-validated in Selenium** — not ported to the Selenium
  suite yet (only 6 of the Playwright suite's tests were ported per
  DEC-002's risk-based selection, predating this bug's discovery).
